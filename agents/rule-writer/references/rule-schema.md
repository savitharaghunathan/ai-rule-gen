# Rule YAML Schema

## Rule File Structure

Each rule YAML file contains a list of rules. Rules are grouped by concern â€” one file per functional area (e.g., `web.yaml`, `ejb.yaml`, `security.yaml`).

```yaml
- ruleID: spring-boot-3-to-spring-boot-4-00010
  description: Replace @RequestMapping with specific annotations
  category: mandatory
  effort: 3
  labels:
    - konveyor.io/source=spring-boot-3
    - konveyor.io/target=spring-boot-4
    - konveyor.io/generated-by=ai-rule-gen
    - konveyor.io/test-result=untested
    - konveyor.io/review=unreviewed
  message: |
    Replace @RequestMapping with specific HTTP method annotations.

    ## Before
    ```java
    @RequestMapping(value = "/api", method = RequestMethod.GET)
    ```

    ## After
    ```java
    @GetMapping("/api")
    ```
  links:
    - title: Spring Boot 4 Migration Guide
      url: https://spring.io/blog/migration
  when:
    java.referenced:
      pattern: org.springframework.web.bind.annotation.RequestMapping
      location: ANNOTATION
```

## Required Fields

| Field | Type | Required | Description |
|---|---|---|---|
| `ruleID` | string | yes | Unique identifier. Format: `<source>-to-<target>-NNNNN` (5-digit, increments of 10). Generated by `go run ./cmd/construct` |
| `when` | Condition | yes | The matching condition. See condition-types.md. Must not be empty |
| `message` or `tag` | string / string[] | one required | `message` for migration guidance with Before/After examples; `tag` for tagging-only rules. At least one must be set |

## Optional Fields

| Field | Type | Description |
|---|---|---|
| `description` | string | Short description (~120 chars max). Shown in summary views. Auto-truncated by `go run ./cmd/construct` if the pattern rationale is longer |
| `category` | string | `mandatory`, `optional`, or `potential`. See below |
| `effort` | int | Migration effort estimate, 1-10. See complexity-to-effort mapping below |
| `labels` | string[] | `konveyor.io/<key>=<value>` labels. Auto-populated by `go run ./cmd/construct` |
| `links` | Link[] | Documentation references. Each has `url` (required) and `title` |
| `customVariables` | CustomVariable[] | Extract variables from matched code. Fields: `pattern` (regex), `name`, `defaultValue`, `nameOfCaptureGroup` |
| `tag` | string[] | Tags to apply (alternative to `message` for tag-only rules) |

## Categories

| Category | Meaning | When to Use |
|---|---|---|
| `mandatory` | Code will not compile, deploy, or function without this change | API removals, namespace renames, breaking behavioral changes |
| `optional` | Code works without the change, but the change is recommended | Deprecated API usage, performance improvements, best practice updates |
| `potential` | May or may not need to change depending on how the API is used | Context-dependent changes, APIs with partial compatibility |

## Effort / Complexity Mapping

`go run ./cmd/construct` converts human-readable complexity (from patterns.json) to numeric effort:

| Complexity (in patterns.json) | Effort (in rule YAML) | Typical Scenario |
|---|---|---|
| `trivial` | 1 | Simple rename, import swap, annotation swap |
| `low` | 3 | Straightforward API change, 1-2 line fix |
| `medium` | 5 | Moderate refactoring, method signature changes |
| `high` | 7 | Significant redesign, new patterns required |
| `expert` | 9 | Architectural change, multiple files affected |

Default (if complexity is unrecognized): effort 5.

## Labels

Labels follow `konveyor.io/<key>=<value>` format. `go run ./cmd/construct` automatically adds these initial labels to every rule:

| Label | Value | Description |
|---|---|---|
| `konveyor.io/source` | from patterns.json `source` | Source technology being migrated from |
| `konveyor.io/target` | from patterns.json `target` | Target technology being migrated to |
| `konveyor.io/generated-by` | `ai-rule-gen` | Marks rule as AI-generated |
| `konveyor.io/test-result` | `untested` â†’ `passed`/`failed` | Set by `go run ./cmd/stamp` after kantra tests |
| `konveyor.io/review` | `unreviewed` | For human review tracking |

Multiple source/target labels can exist for rules that apply to multiple migration paths.

The `stamp` command updates `konveyor.io/test-result` from `untested` to `passed` or `failed` based on kantra results.

## Message Format

Messages should be concise (2-4 sentences), actionable, and include Before/After code examples when possible:

```
Replace deprecated javax.servlet API with the Jakarta EE equivalent. The javax.servlet 
namespace was renamed to jakarta.servlet in Jakarta EE 9+.

## Before
```java
import javax.servlet.http.HttpServlet;
```

## After
```java
import jakarta.servlet.http.HttpServlet;
```
```

Key rules for messages:
- Explain WHAT needs to change and WHY
- State the replacement if applicable
- Include Before/After code examples as markdown code blocks with language syntax highlighting
- No headers or labels wrapping the message â€” just the text content itself
- If a custom `message` is provided in patterns.json, it is used as-is. If empty, auto-generated from `source_pattern` + `rationale`

## Ruleset Format

Each rules directory needs a `ruleset.yaml` file:

```yaml
name: spring-boot-4/spring-boot-3
description: Rules for migrating from spring-boot-3 to spring-boot-4
labels:
  - konveyor.io/source=spring-boot-3
  - konveyor.io/target=spring-boot-4
```

- `name` format: `<target>/<source>` (generated by `go run ./cmd/construct`)
- `description`: auto-generated as "Rules for migrating from `<source>` to `<target>`"
- `labels`: source + target labels

## Validation Rules

`go run ./cmd/validate` checks:

**Errors (rule fails validation):**
- `ruleID` missing, contains newlines/semicolons, or is duplicated
- Neither `message` nor `tag` is set
- `when` condition is empty (no condition type set)
- `category` is present but not one of: `mandatory`, `optional`, `potential`
- Condition-specific required fields missing:
  - `java.referenced`: `pattern` required; `location` must be valid if present (14 valid values)
  - `java.dependency`: `name` or `nameRegex` required
  - `go.referenced`: `pattern` required
  - `go.dependency`: `name` or `nameRegex` required
  - `nodejs.referenced`: `pattern` required
  - `csharp.referenced`: `pattern` required; `location` must be valid if present (4 valid values: ALL, METHOD, FIELD, CLASS)
  - `builtin.filecontent`: `pattern` required; pattern and filePattern must be valid regex
  - `builtin.file`: `pattern` required
  - `builtin.xml`: `xpath` required
  - `builtin.json`: `xpath` required
  - `builtin.xmlPublicID`: `regex` required and must be valid regex
- `or`/`and` combinator children are also validated recursively

**Warnings (rule passes but flagged):**
- `effort` outside 1-10 range
- `konveyor.io/source` or `konveyor.io/target` label present but missing a value
